## 1. Architecture Overview

### Modules

Five explicit modules replace the current monolithic host + Docker Compose setup:

```
┌─────────────────────────────────────────────────────────────────────┐
│  HOST  (Ansible: 4-5 roles, bare metal only)                        │
│  • Docker Engine                                                     │
│  • Tailscale (network layer)                                         │
│  • systemd (unit for compose stack auto-start)                       │
│  • unattended-upgrades (security patches)                            │
│  • workstation (docker-compose up on boot)                           │

    │                                                      │

│  AUTOMATION      │     │  PAI SESSION CONTAINERS         │
```

### Module Boundary: PAI vs Imladris-Additive

| Module | Origin | What it does |
|--------|--------|-------------|
| PAI session container | **PAI upstream** | Claude Code runtime, MEMORY, .claude config, skills |
| Windmill automation stack | **Imladris additive** | Triage pipeline, investigation scripts, cron jobs |

| Host Ansible roles | **Imladris additive** | Docker, Tailscale, hardening, workstation bootstrap |
| CloudFormation templates | **Imladris additive** | EC2, EBS, networking (AWS-specific overlay) |

## 2. Per-Session PAI Container Model

### Decision: Per-session containers win

**Rationale (from Council):** Single shared container is operationally simpler but optimizes the wrong failure mode. Per-session isolation prevents:
- Session A's half-applied tool outputs corrupting Session B's working state
- Cross-session blast radius from a compromised context
- Concurrent .claude file mutations without locking

**Session lifecycle:**
```
pai start <name>          # creates/starts named container, attaches tmux
pai attach <name>         # docker exec -it pai-{name} tmux attach -t main
pai list                  # docker ps --filter label=pai-session
pai stop <name>           # sends tmux detach, container stays alive (-dit)
pai destroy <name>        # merges MEMORY deltas, then docker rm
```

### Container spec

```dockerfile
FROM node:20-slim

# System packages
RUN apt-get update && apt-get install -y \
    tmux git curl jq fzf ripgrep \
    python3 python3-pip \
    && rm -rf /var/lib/apt/lists/*

# Claude Code CLI
RUN npm install -g @anthropic-ai/claude-code

# Session entrypoint
COPY entrypoint.sh /usr/local/bin/entrypoint.sh
RUN chmod +x /usr/local/bin/entrypoint.sh

ENTRYPOINT ["/usr/local/bin/entrypoint.sh"]
```

**entrypoint.sh:**
```bash
#!/bin/bash
# Secrets: already injected as env vars via --env-file from tmpfs at start time
# Clear API key from environment after Claude Code picks it up (process inherits a cleared env)
exec tmux new-session -s main -d
exec tmux attach -t main
```

### Docker run flags (required)
```bash
docker run -dit \
  --name pai-${SESSION_NAME} \
  --label pai-session=true \
  --label pai-session-name=${SESSION_NAME} \
  --network imladris_net \
  --env-file /run/secrets/pai-env \         # tmpfs path, written by session script at start
  --mount type=volume,src=pai-config,dst=/home/node/.claude,readonly \
  --mount type=volume,src=pai-memory,dst=/home/node/.claude/MEMORY \
  --mount type=volume,src=pai-session-${SESSION_NAME},dst=/home/node/work \
  --mount type=bind,src=/pai/inbox,dst=/pai/inbox,readonly \
  --mount type=bind,src=/pai/outbox,dst=/pai/outbox \
  --cap-drop=ALL \
  --security-opt=no-new-privileges \
  --memory=4g \
  --cpus=2 \
  ghcr.io/sethdf/imladris/pai:latest
```

**Critical: NO Docker socket mount.** The pai container never gets access to `/var/run/docker.sock`. Attack chain: prompt injection → shell command → `docker run --privileged` → host escape. There is no use case that justifies this risk.

## 3. Volume Strategy

### Volume taxonomy

| Volume name | Type | Access | What lives there |
|-------------|------|--------|-----------------|
| `pai-config` | Named Docker volume | **Read-only** in session containers | .claude/settings.json, CLAUDE.md, skills/, hooks/, agents/ |
| `pai-memory` | Named Docker volume | **Read-write** (one writer at a time) | ~/.claude/MEMORY/ — MEMORY files, PRDs, History |
| `pai-session-{name}` | Named Docker volume | **Read-write** (exclusive) | Per-session working directory (repos, scratch) |
| `windmill-db` | Named Docker volume | **Read-write** (postgres only) | Windmill Postgres database |
| `/pai/inbox` | Host bind mount (ro in PAI) | Read by PAI, write by Windmill | Windmill→PAI job requests |
| `/pai/outbox` | Host bind mount (rw in PAI) | Write by PAI, read by Windmill | PAI→Windmill job results |

### MEMORY persistence contract

| Event | What survives | What's destroyed |
|-------|--------------|-----------------|
| Container detach (tmux detach) | Everything in pai-memory and pai-config | Nothing |
| Container stop (docker stop) | Everything in named volumes | Nothing |
| Container destroy (docker rm) | pai-memory, pai-config | `pai-session-{name}` workspace |
| Host reboot | All named volumes (Docker volumes survive reboots) | Nothing |
| Host migration | Named volumes (exported via docker save/docker volume export) | Host-local bind mounts |

### MEMORY concurrent write strategy

MEMORY volume is **append-only** during sessions. On session destroy:
1. Session container writes new/modified MEMORY files to a staging path: `/memory-staging/`
2. Session manager script runs `rsync --append-verify` from staging to canonical `pai-memory` volume
3. Staging volume is destroyed

For simultaneous sessions (rare, single user): merge is serialized by the session manager via a lockfile (`/pai-memory-merge.lock`).

### Volume backup/migration (host portability)

```bash
# Backup
docker run --rm \
  -v pai-config:/source:ro \
  -v $(pwd)/backups:/dest \
  alpine tar czf /dest/pai-config-$(date +%Y%m%d).tar.gz -C /source .

# Restore on new host
docker volume create pai-config
docker run --rm \
  -v pai-config:/dest \
  -v $(pwd)/backups:/source:ro \
  alpine tar xzf /source/pai-config-20260101.tar.gz -C /dest

# Or use: docker volume export (Docker 26+)
```

## 4. Session Manager Design

### Decision: Shell script (not DevPod)

DevPod solves developer environment provisioning, not session reconnect. We need `docker run/exec/stop/rm` with named containers, tmux, and a merge-on-destroy step. 50 lines of bash is the right tool.

**`pai` shell script (lives at `/usr/local/bin/pai` on host):**

```bash
#!/bin/bash
set -euo pipefail

SESSION_NAME="${2:-default}"
CONTAINER_NAME="pai-${SESSION_NAME}"
IMAGE="ghcr.io/sethdf/imladris/pai:latest"
ENV_FILE="/run/pai-secrets/env"  # tmpfs, written by `pai start`

cmd_start() {
  # Fetch secrets from BWS into tmpfs
  mkdir -p /run/pai-secrets && chmod 700 /run/pai-secrets
  mount -t tmpfs tmpfs /run/pai-secrets  # ephemeral
  bws secret get anthropic_api_key | jq -r '.value' | \
    xargs -I{} echo "ANTHROPIC_API_KEY={}" > "${ENV_FILE}"
  bws secret get github_pat | jq -r '.value' | \
    xargs -I{} echo "GITHUB_TOKEN={}" >> "${ENV_FILE}"
  chmod 600 "${ENV_FILE}"

  if docker ps -a --format '{{.Names}}' | grep -q "^${CONTAINER_NAME}$"; then
    docker start "${CONTAINER_NAME}"
  else
    docker run -dit \
      --name "${CONTAINER_NAME}" \
      --label pai-session=true \
      --label pai-session-name="${SESSION_NAME}" \
      --network imladris_net \
      --env-file "${ENV_FILE}" \
      --mount type=volume,src=pai-config,dst=/home/node/.claude,readonly \
      --mount type=volume,src=pai-memory,dst=/home/node/.claude/MEMORY \
      --mount type=volume,src="pai-session-${SESSION_NAME}",dst=/home/node/work \
      --mount type=bind,src=/pai/inbox,dst=/pai/inbox,readonly \
      --mount type=bind,src=/pai/outbox,dst=/pai/outbox \
      --cap-drop=ALL \
      --security-opt=no-new-privileges \
      --memory=4g --cpus=2 \
      "${IMAGE}"
  fi
  docker exec -it "${CONTAINER_NAME}" tmux attach -t main || \
    docker exec -it "${CONTAINER_NAME}" tmux new-session -s main
}

cmd_attach() {
  docker exec -it "${CONTAINER_NAME}" tmux attach -t main
}

cmd_list() {
  docker ps --filter label=pai-session=true \
    --format "table {{.Names}}\t{{.Status}}\t{{.RunningFor}}"
}

cmd_destroy() {
  # Merge MEMORY before destroy
  docker exec "${CONTAINER_NAME}" bash -c \
    "rsync -a ~/.claude/MEMORY/ /memory-staging/ 2>/dev/null || true"
  # Acquire merge lock
  (flock 9; rsync --append-verify /memory-staging/ /pai/memory/; ) 9>/pai-memory-merge.lock
  docker rm -f "${CONTAINER_NAME}"
  docker volume rm "pai-session-${SESSION_NAME}" 2>/dev/null || true
}

case "${1:-help}" in
  start)   cmd_start ;;
  attach)  cmd_attach ;;
  list)    cmd_list ;;
  destroy) cmd_destroy ;;
  *) echo "Usage: pai {start|attach|list|destroy} [session-name]" ;;
esac
```

### Session naming convention

```
pai-{slug}                          # named session (persists)
pai-default                         # default single session
pai-imladris-docs                   # project-scoped session
pai-{YYYYMMDD}-{topic}              # date-scoped session
```

### Session auto-resume on host reboot

A systemd unit on the host restarts all stopped pai-* containers:

```ini
[Unit]
Description=Resume PAI sessions on boot
After=docker.service

[Service]
Type=oneshot
ExecStart=/bin/bash -c 'docker ps -a --filter label=pai-session=true --filter status=exited --format "{{.Names}}" | xargs -r docker start'
RemainAfterExit=yes

[Install]
WantedBy=multi-user.target
```

### Container cleanup (orphan reaper)

Containers run with `--rm` when possible, but for named sessions (which must persist), the session manager script owns lifecycle. A cron fallback prunes exited containers older than 24 hours:

```bash
# /etc/cron.d/pai-prune
0 4 * * * root docker container prune --filter label=pai-session=true --filter until=24h --force
```

## 5. Windmill↔PAI Boundary

### Decision: Filesystem inbox/outbox queue

Windmill writes job payloads to `/pai/inbox/`. PAI session watches via inotifywait and processes them. Results written to `/pai/outbox/`. Windmill polls outbox.

**Why not HTTP API:** Adds a daemon requirement and network layer for a local IPC problem. Filesystem is simpler, auditable, and survives container restarts.

**Security requirements (Rook's conditions for this to be safe):**
1. `/pai/inbox` write permission restricted to Windmill worker UID only (`chown windmill:windmill`, `chmod 770`)
2. Every message schema-validated by PAI before processing (JSON Schema validation, reject unknown fields)
3. Full audit log of all inbox/outbox messages written to a separate append-only log volume

**Message format:**
```json
{
  "job_id": "uuid",
  "source": "windmill",
  "type": "knowledge_query | investigation_context | triage_input",
  "payload": { ... },
  "created_at": "ISO8601"
}
```

**PAI inbox watcher (inside session container):**
```bash
inotifywait -m /pai/inbox -e create -e moved_to |
  while read dir event file; do
    [[ "$file" == *.json ]] || continue
    # Schema validate before processing
    if jq -e 'has("job_id") and has("type") and has("payload")' "/pai/inbox/$file" > /dev/null; then
      process_pai_job "/pai/inbox/$file"
    else
      echo "REJECTED: invalid schema" >> /pai/inbox/rejected.log
    fi
    mv "/pai/inbox/$file" "/pai/inbox/processed/"
  done
```

### Windmill workers access to PAI knowledge base

Windmill scripts continue to use the MCP server (`mcp_server.ts`) which exposes investigation tools. For PAI-specific knowledge (MEMORY files, domain knowledge), Windmill queries the outbox or calls a lightweight read-only HTTP endpoint exposed by the PAI session container on the Docker network (`pai-default:8099/knowledge`).

## 6. Secrets Injection

### Decision: BWS runtime fetch → tmpfs → env-file → cleared after use

**Pattern:**
1. Host: `bws` CLI fetches secrets at session start time
2. Written to tmpfs at `/run/pai-secrets/env` (mode 600)
3. Passed to container via `--env-file` flag
4. Container entrypoint clears secrets from environment after SDK initialization
5. Tmpfs is ephemeral — disappears on host reboot

**Never:**
- Env vars baked into images
- Secrets in Docker volumes (persistent, snapshotted)
- `docker inspect` readable env vars (--env-file bypasses this for the file, but env vars in the container are still visible to `docker inspect` — entrypoint should unset them after use)

### Secrets inventory

| Secret | BWS key | Where it goes |
|--------|---------|--------------|
| ANTHROPIC_API_KEY | `anthropic_api_key` | PAI container env (cleared after Claude Code init) |
| BWS_ACCESS_TOKEN | `bws_access_token` | Host only, never in container |
| GITHUB_TOKEN | `github_pat` | PAI container env |
| AWS_ACCESS_KEY_ID / SECRET | `aws_*` | PAI container env (for AWS CLI inside container) |
| TAILSCALE_AUTH_KEY | `tailscale_auth` | Host only, Ansible only |

### GitHub PAT injection

Same pattern as ANTHROPIC_API_KEY: fetched by host BWS at session start, injected via env-file, used for `git` operations inside container. Never stored in image or volume.

## 9. CloudFormation Scope (Host-Agnostic)

### What CloudFormation manages (AWS-specific overlay only)

CloudFormation continues to manage AWS-specific resources:
- EC2 instance (instance type, IAM role, security groups)
- EBS volume for Docker volume storage (named volumes backed by EBS)
- VPC/subnet/security groups (no inbound, Tailscale only)
- S3 bucket for volume backups
- KMS key for EBS encryption

### What CloudFormation does NOT manage

- Everything inside Docker (volumes, containers, compose stack)
- Tailscale configuration (managed by Ansible)
- Host OS state (managed by Ansible)

### Host-agnostic operation

On a laptop (non-AWS):
- No CloudFormation needed
- Docker Desktop replaces Docker Engine
- Volumes stored on local filesystem
- Same `docker-compose.yml` and `pai` shell script work identically
- Only difference: no NVMe Ansible role, no EBS volume

## 10. NVMe Usage in Docker-Modular World

NVMe instance store (m7gd.xlarge provides ~950GB NVMe) is used as Docker's data root:

```json
// /etc/docker/daemon.json
{
  "data-root": "/nvme/docker",
  "log-driver": "json-file",
  "log-opts": { "max-size": "50m", "max-file": "3" }
}
```

All named volumes (pai-config, pai-memory, pai-session-*, windmill-db) live on NVMe. This gives fast I/O for MEMORY operations and Postgres. EBS provides durability for backups only.

**Ansible nvme role:** Formats NVMe at instance start, mounts at `/nvme`, configures Docker daemon to use it. On host migration, volumes are restored from S3/EBS backup.

### Strategy: GitHub Actions → GHCR

```yaml
# .github/workflows/build-pai-image.yml
on:
  push:
    branches: [main]
    paths: ['docker/pai/**']

jobs:
  build:
    runs-on: ubuntu-latest

      - uses: actions/checkout@v4
      - uses: docker/login-action@v3

          registry: ghcr.io

        with:
          context: docker/pai
```

### Upgrade path

New PAI version → rebuild image → `docker pull ghcr.io/sethdf/imladris/pai:latest` on host → `pai destroy old-session && pai start new-session`. MEMORY volume survives. Config volume survives. Workspace volume is typically recreated.

## 17. Migration Path

### Phase 0 — NOW (no architecture change required)

**Fix the live .claude:rw vulnerability:**

```yaml
# docker-compose.yml — change all windmill worker mounts
# FROM:
volumes:
  - ~/.claude:/home/node/.claude:rw
# TO:
volumes:
  - ~/.claude:/home/node/.claude:ro
```

This is an immediate mitigation. All Windmill workers currently have write access to the PAI brain (all MEMORY files, steering rules, hooks). Making this read-only closes the prompt injection write vector NOW.

### Phase 1 — Docker-ified PAI sessions (lowest risk, highest value)

1. Build PAI container image (`docker/pai/Dockerfile` + CI)
2. Create named volumes: `pai-config`, `pai-memory`
3. Migrate current `~/.claude` content into volumes
4. Install `pai` session manager script on host
5. Test `pai start default` → attach → tmux works → detach → reattach
6. Validate MEMORY persistence across container destroy/recreate
7. Collapse Ansible to 5 roles (base, docker, tailscale, workstation, nvme)
8. Keep Windmill Compose stack unchanged (no migration)
9. Keep MCP servers on host unchanged

**Risk:** Low. Windmill stack untouched. Host-to-container migration of Claude Code is well-proven by Anthropic devcontainer.

## 18. Anti-Recommendations (What NOT To Do)

1. **Never mount the Docker socket into PAI containers.** Attack chain: prompt injection → shell execution → `docker run --privileged --pid=host -v /:/host alpine chroot /host`. Unconditional prohibition.

2. **Never use env vars for secrets in `docker run`.** They appear in `docker inspect`, `/proc/1/environ`, and any crash dump. Use `--env-file` pointing to tmpfs, cleared after container init.

3. **Do not use DevPod.** It solves developer environment provisioning, not session reconnect. A 50-line shell script owns the behavior, has no upgrade surface, and is readable by anyone debugging the system.

4. **Do not try to Dockerize Windmill workers and PAI simultaneously in Phase 1.** The migration complexity is additive. Each boundary change is an independent risk. Phase 1 is PAI only.

5. **Do not burn the Ansible bridge before the container path is proven.** Keep the existing 16-role Ansible working until `pai start default` is stable through at least one host reboot. Rollback must be under 10 minutes.

6. **Do not share per-session workspace volumes.** Each session gets exclusive ownership of its workspace. Cross-session file access goes through the canonical repo (git push/pull), not volume sharing.

7. **Do not use Docker Swarm secrets.** Adds Swarm orchestration complexity for a single-node setup. tmpfs + env-file is sufficient and simpler.

8. **Do not skip schema validation on inbox queue messages.** Without it, the inbox/outbox queue is a filesystem socket that any process writing to `/pai/inbox` can use to inject instructions into PAI reasoning. Schema validation is the trust boundary.
